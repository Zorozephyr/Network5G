# 1. Current Architectural Issues with O-RAN xApps

The Open Radio Access Network (O-RAN) architecture relies on the Near-Real-Time RAN Intelligent Controller (Near-RT RIC) to host "xApps" — microservices that execute control-plane optimizations (e.g., traffic steering, scheduling) within a strict 10ms to 1s window. Currently, xApps are deployed as standard Docker containers orchestrated by Kubernetes. 

A rigorous analysis of recent research (2024–2026) reveals four critical structural flaws in this container-based approach that prevent the realization of true near-real-time control. Problems 1.3 and 1.4 are extensively documented in the literature with CVE-level evidence and active research tracks. Problems 1.1 and 1.2, while architecturally self-evident and acknowledged by systems like CORMO-RAN and the O-RAN Alliance's E2AP v7.00 revisions, have not yet been quantified end-to-end in a published benchmark — making their rigorous measurement a contribution opportunity.

## 1.1. Container Overhead and Cold Start Latency

* **The Problem:** Horizontal scaling of xApps incurs severe "cold start" latency. A standard xApp container includes a full Linux base image, language runtime (Python/Java), RMR libraries, and ML weights, resulting in a 200MB–1GB image. The composite startup path involves: container image pull and instantiation (seconds), RMR registration and route establishment (sub-second), E2 subscription establishment (seconds), and ML model initialization (seconds to minutes for large models).
* **The Impact:** For a 10ms control loop, a multi-second startup delay is catastrophic. During this cold start window, the RAN scheduler runs blind, defaulting to baseline heuristics which degrades URLLC (Ultra-Reliable Low-Latency Communication) slice performance.
* **Evidence Assessment:**
    * **Documented (component-level):** Studies have measured 22–38ms for lightweight xApps (8.5–16MB images) and up to 5 minutes for ML model stabilization periods. Standard Kubernetes pod startup on edge hardware is measured in hundreds of milliseconds to seconds without pre-warming. AI-driven proactive autoscaling frameworks explicitly target this bottleneck.
    * **Documented (implicit acknowledgment):** CORMO-RAN's entire motivation assumes that xApp restarts cause state loss and degraded control quality. WA-RAN demonstrates live Wasm plugin swaps within 5G slot durations (1000µs), positioning Wasm's instantiation speed as a key advantage. BubbleRAN's ultra-lean architecture design philosophy is partly driven by minimizing startup overhead.
    * **Not yet documented (gap):** No published work benchmarks the full end-to-end cold start of a production xApp (image pull + container init + RMR registration + E2 subscription + ML model load) on an OSC Near-RT RIC deployment. The composite "2 to 30 seconds" estimate is inferred from individual component measurements, not a single measured value.
* **Research Context:** Frameworks like *OREO* (INFOCOM 2024) reduce cold starts by sharing xApp instances across services. *MANATEE* (arXiv 2601.14009) uses canary deployments to avoid full-cutover restarts. The serverless/FaaS community extensively documents container cold start as a general problem, and WA-RAN (HotNets '24) explicitly positions Wasm's microsecond instantiation as the solution for RAN workloads.

## 1.2. E2 Subscription Disruption (No Hitless Upgrade)

* **The Problem:** E2 subscriptions are heavily stateful. The gNB sends E2 Indication (KPM) reports tagged with a specific Subscription ID that routes through the RIC Message Router (RMR) to a specific xApp instance. When a K8s rolling update occurs, the old pod is terminated.
* **The Impact:** To avoid E2 Terminator (E2T) crashes from dangling states, the old pod must gracefully delete its E2 subscription before dying. The new pod must cold-start and establish a brand new subscription. This tear-down and rebuild process creates a gap of zero RAN control during which no E2 Indications are processed and no E2 Control actions are issued.
* **Evidence Assessment:**
    * **Documented (mechanism):** The O-RAN SC Subscription Manager documentation explicitly describes the teardown/re-establishment lifecycle, including RMR route invalidation during pod IP changes, subscription merge failures during overlapping pod transitions, and timer-based expiration of stale subscriptions. The SubMgr uses timers to supervise subscription requests — a rolling update that exceeds these timeouts triggers subscription deletion.
    * **Documented (system-level acknowledgment):** *CORMO-RAN* (arXiv 2506.19760) is built entirely around this problem. It distinguishes between Stateful Migration (SM, with explicit downtime T_D > 0) and SDL-based migration (T_D = 0, but requires xApps to be pre-architected for SDL externalization). The paper states that SDL-based approaches "introduce periodic service disruptions" under high cluster load. The O-RAN Alliance added "RIC Subscription State Control" to E2AP v7.00 (April 2025) — a standardization-level acknowledgment that hitless E2 subscription handling was missing from prior specifications.
    * **Not yet documented (gap):** No published work measures the actual end-to-end duration of the "zero RAN control" gap during a standard xApp rolling update on the OSC Near-RT RIC. The gap duration depends on container startup time, SubMgr timer values, and E2 subscription negotiation latency — all deployment-specific. The previously stated "5–60 second" range is an architectural inference, not a measured value.
* **Research Context:** *CORMO-RAN* proposes container-level migration with two strategies (SM-MR minimizing resources, SM-MD minimizing downtime). *MANATEE* mitigates risk via progressive canary deployment but does not address E2 subscription state transfer. The O-RAN standard's designated solution (SDL-based state externalization) requires xApps to be architecturally designed for it — a requirement that most existing xApps do not meet, especially those with large ML models stored in process memory (acknowledged in Section 11.3 of the cross-cutting analysis).

## 1.3. Security Vulnerabilities and Lack of Isolation

* **The Problem:** xApps run as standard Linux containers. They share the kernel, network namespaces, and rely on cgroups for basic resource limitation. This provides no defense against application-layer exploits.
* **The Impact:** A malicious or poorly written xApp can crash the entire RIC. Specifically, **CVE-2023-41628** documents how an xApp sending out-of-order E2 subscription responses instantly crashes the E2T component, causing a Denial of Service (DoS) for the entire RAN control plane. **CVE-2023-42358** demonstrates unauthorized REST API access across xApp boundaries. More recently, **CVE-2024-34046/34047/34048** (O-RAN I-Release) document crashes in E2T and e2mgr from missing array size checks and subscription counter failures. Furthermore, compromised xApps can exfiltrate sensitive UE telemetry via shared SDL access.
* **Evidence Assessment:**
    * **Documented (CVE-level):** Hung et al. (IEEE Open Journal of Communications, 2024) performed a systematic CVE-level security analysis of the OSC H-Release, discovering that "the existing near-RT RIC does NOT enforce the access control permissions specified by O-RAN Alliance WG11." Multiple CVEs filed (2023–2025). CVE-2025-67398 and CVE-2025-52150 are the most recent disclosures, documenting remote DoS via crafted xApp control messages.
    * **Documented (industry reports):** Trend Micro's *"Attack of the xApps"* report explicitly cites "lack of sandboxing/isolation" as the primary enabler for xApp-driven attacks.
    * **Documented (standardization):** O-RAN WG11 published a Zero Trust Architecture white paper (2024), acknowledging that current compliance is at "Initial" level with "Advanced" remaining aspirational.
    * **Documented (research systems):** *eZTrust/OZTrust* (University of Utah) provides eBPF-based packet-level enforcement as a retrofit solution. The system demonstrates blocking unauthorized cross-xApp data access and E2 message injection with negligible overhead. However, it operates at the network packet level — xApps still possess OS-level network stacks and the enforcement is monitoring-based rather than structural.
* **Research Context:** This is the most well-documented of the four problems. The combination of filed CVEs, an industry security report, a WG11 white paper, and multiple research systems (eZTrust, ZTRAN, ZT-RIC) targeting this specific issue provides strong evidence that container-based xApp isolation is inadequate for a production multi-tenant Near-RT RIC.

## 1.4. Multi-Vendor Conflicts and the Enforcement Gap

* **The Problem:** In a multi-vendor RIC, Vendor A's Traffic Steering xApp and Vendor B's Energy Saving xApp may issue conflicting E2 Control commands for the same Physical Resource Blocks (PRBs). The O-RAN Alliance defines three conflict types: *direct* (same parameter), *indirect* (different parameters affecting the same KPI), and *implicit* (cascading KPI-triggered actions). PACIFISTA demonstrates 16–30% performance degradation from such conflicts.
* **The Impact:** Direct conflicts cause "parameter flipping," where the gNB rapidly oscillates between configurations. Indirect and implicit conflicts produce subtler degradation that is harder to detect and attribute. 
* **Evidence Assessment:**
    * **Documented (extensively):** This is the most actively researched problem in the Near-RT RIC literature. *PACIFISTA* (arXiv 2405.04395) provides statistical profiling and severity analysis. *COMIX* (arXiv 2501.14619) uses Network Digital Twins for DRL-specific conflict resolution. *Adamczyk CMF* (IEEE ComMag 2023) proposes standards-aligned detection/resolution within the RIC. *Zolghadr et al.* (IEEE WCNC 2025) use GraphSAGE GNNs to discover hidden indirect/implicit conflicts. Multi-agentic LLM systems (arXiv 2603.07375) achieve 70% improvement in deployment accuracy for conflict-aware orchestration. The O-RAN Alliance WG3 published a Conflict Mitigation Technical Specification (2024) formally mandating a Conflict Mitigation Function.
    * **The unsolved enforcement gap:** All existing conflict management systems — PACIFISTA, COMIX, Adamczyk CMF, GraphSAGE, LLM agents — operate at the detection, resolution, or prevention layers. They assume that xApps faithfully route their E2 Control messages through the designated Conflict Manager via RMR. However, in a container-based RIC, this assumption relies on the honor system: xApps possess full Linux network stacks and shared kernel namespaces. A compromised or malicious xApp can bypass the RMR/CM entirely and inject E2 Control messages directly to the E2 Terminator via the host network, rendering all upstream conflict resolution logic ineffective. *eZTrust/OZTrust* partially addresses this with eBPF packet filtering, but the xApp still possesses the OS-level capability to attempt bypass — enforcement is reactive (drop unauthorized packets) rather than structural (remove the capability entirely).
* **Research Context:** The conflict detection and resolution problem is well-addressed by existing research. The critical gap is the enforcement layer: ensuring that conflict management decisions are tamper-proof and cannot be circumvented by a compromised xApp. No existing container-based system provides structural guarantees that an xApp cannot bypass its designated communication channel.
